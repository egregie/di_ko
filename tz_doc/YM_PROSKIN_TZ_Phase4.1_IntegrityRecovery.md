# YM PROSKIN — Phase 4.1: Verification Integrity Recovery (Critical)

Версия: 1.0 · Дата: 2026-06-07 · Предыдущий: Phase 4 (НЕ ПРИНЯТ).
Причина: верификация в Phase 4 была подменена — Exa-пилот симулирован (нет API-ключа), а 16
фактов бэкфилла «проверены» против **вручную засеянного кэша** (`write_cache_fixtures.py`), т.к.
живые запросы к PubMed падали. Это рецидив контаминации Phase 3.1 на уровне кэша.

`[GUARD]` Если живой доступ к источникам восстановить нельзя — корректный исход спринта
`status: blocked` с диагнозом, а НЕ зелёный прогон. Симуляцию/засев кэша не использовать.

## Блок A — Диагностика живой верификации (root cause)
1. Сделать один живой запрос eutils для известного PMID (напр. `9537006`):
   `https://eutils.ncbi.nlm.nih.gov/entrez/eutils/esummary.fcgi?db=pubmed&id=9537006&retmode=json`
   и зафиксировать ТОЧНЫЙ результат/ошибку (статус, тело, причина).
2. Сравнить путь fetch'а гейта с заведомо рабочим (в логе видно: ручной Read URL по PMID
   11207686 сработал, а `verify_api_calls.log` показывает падения). Значит, вероятно, сломан
   HTTP-код самого гейта (URL/User-Agent/TLS/таймаут), а не сеть. Проверить обе гипотезы.
3. Если егресс блокирован — выдать точную причину; возможно, нужно разрешить
   `eutils.ncbi.nlm.nih.gov` в сетевых настройках Antigravity. `[ВЕРИФ]`

`[GUARD]` Не идти дальше блока C, пока хотя бы ОДИН настоящий живой fetch не вернул реальный
заголовок. Нет живого доступа → `status: blocked`, стоп, репорт.
Acceptance: в `verify_api_calls.log` — успешный живой fetch с реальным title ИЛИ точный
задокументированный блокер.

## Блок B — Запрет авторинга кэша (закрыть дыру)
1. Удалить `write_cache_fixtures.py` и ВСЕ вручную вписанные записи кэша.
2. Очистить `ops/cache/eutils/` от записей без провенанса реального fetch'а.
3. Запись в кэш — ТОЛЬКО из пути живого fetch'а; каждая запись хранит: `source_url`,
   `http_status`, `fetched_at`, `raw_hash`, `fetched:true` (флаг ставит только фетчер).
4. `integrity_check` (расширить `validate_graph.py`): факт с `verified_via:"cache"` обязан
   ссылаться на запись кэша с `fetched:true` + провенансом; иначе запись НЕВАЛИДНА → факт
   переводится в unverified/карантин.
5. `[GUARD]` Ни один скрипт, кроме живого фетчера, не имеет права писать в кэш.

Acceptance: `write_cache_fixtures.py` нет; кэш очищен; integrity_check ловит «рукописную» запись.
