## 4. БЛОК D — НАПОЛНЕНИЕ ТЕКСТОМ (граф + авторские рамки)

### D1. Источник текста — два явно разделённых класса
1. **Фактические утверждения** → строго из графа, каждое трассируется к `fact_id` (P018). Текст факта = его `statement` (или аккуратная редактура без изменения смысла).
2. **Авторские рамки** → заголовки, названия осей фреймворка, категории классификации, переходы, структурные подписи. НЕ претендуют на доказанность, фактических утверждений не содержат.

[GUARD] КРИТИЧНО (ядро проекта): на слайде каждый блок, делающий фактическое утверждение (число, эффект, механизм), несёт привязку к fact_id И **grade-маркер** (A/B/C). Авторские рамки — нейтральны, без grade. Зритель должен различать «это доказано (и насколько)» vs «это способ организации мысли».

### D2. Grade-честность на слайде (прямое продолжение аудита 8.7b)
- C-grade факт (их 31 — обзор/неконтролируемое/самооценка) подаётся с рамкой уровня: caption «обзор» / «неконтролируемое, n=N» / «самооценка». НЕ как аксиома.
- A-grade — может подаваться уверенно (RCT/MA/SR).
- [GUARD] Фреймворк/матрица НЕ имеет права превращать C-факт в выглядящий-как-доказанный пункт. Ячейка с фактом → её grade виден.
- Слайды «Evidence Summary» (C2 в обеих деках) — явно показывают распределение grade. Это выполнение премиум-EBM-обещания, не прячем 31 C.

### D3. Объём текста под layout
- Соблюдать `max_chars` из `layouts_v2.json` (h1≤50, callout≤100, body по брендбуку). Если факт длиннее слота — редактировать без потери смысла или выбрать другой слот.
- [GUARD] Не впихивать факт, не влезающий в слот (старый дефект overflow). Лучше короче и в рамках.

### D4. Трассируемость-артефакт
- Для каждой деки сгенерировать `deck_*_provenance.json`: слайд → блок → fact_id → grade (или «author_frame»). Это доказательство P018 для всей деки.
- [ВЕРИФ] Любой фактический блок без fact_id или без grade = FAIL гейта (см. Блок F).
