# YM PROSKIN — Phase 3.1: Gate Authenticity Audit (Batch 1)

Версия: 1.0 · Дата: 2026-06-07 · Предыдущий: Phase 3 Batch 1 (НЕ принят до этого аудита).
Цель: доказать, что verify-at-write гейт адъюдицирует факты **вживую**, а не отдаёт
предзаписанные вердикты/кэш. Пока это не доказано — масштабирование и рендер заморожены.

Легенда: `[GUARD]` жёсткое ограничение · `[ВЕРИФ]` · `[НЕТ ДАННЫХ]`.

---

## Почему этот спринт
- `rejection_rate: 0%` по 3 темам против 25%+19% в Phase 1 — статистически подозрительно.
- Walkthrough сам указывает причину: вердикты 11 фактов внесены в `KNOWN_VERDICTS` внутри
  `evidence.py`, а кэш PubMed предзаполнен. Это превращает верификатор в answer-key:
  «вопрос» и «ответ» написаны одной рукой.
- Доверять графу можно, только если гейт проверяет источник из живого API и судит по
  фактическому абстракту. Это нужно доказать на уже загруженных 11 фактах.

---

## Блок A — Деконтаминация библиотеки проверки `[GUARD]`
1. Удалить `KNOWN_VERDICTS` (и любые хардкод-вердикты) из **production-пути** `lib/evidence.py`.
   Верификатор обязан выносить вердикт динамически: fetch источника → assess по абстракту.
2. Тестовые фикстуры (детерминированные кейсы: fake PMID `99999999` → reject и т.п.) допустимы
   ТОЛЬКО в `run_regression.py` (тест-харнес) и НЕ импортируются в живой путь `verify_gate.py`.
3. `[GUARD]` Запрет: любая ветка, возвращающая вердикт по списку ID без обращения к источнику,
   в production-коде недопустима.

---

## Блок B — Перепроверка 11 фактов вживую (обход кэша)
1. Добавить в `verify_gate.py`/`evidence.py` флаг `--force-live` (или `--no-cache`): игнорирует
   `ops/cache/eutils/`, тянет источник заново.
2. Прогнать ВСЕ 11 фактов Batch 1 (`fact_0017`..`fact_0027`) через гейт с `--force-live` и
   отключённым KNOWN_VERDICTS. Для каждого зафиксировать:
   `fact_id | source_id | pmid | live_exists | fetched_title(≤12 слов) | verdict | evidence_ok`.
   `[GUARD]` Вердикт — только по фактически полученному абстракту, не по памяти модели.
3. Выход: `ops/logs/gate_authenticity_batch1.md` + итоговая **реальная** rejection-rate.

---

## Блок C — Реконсиляция
1. Факты с `SOURCE_NOT_FOUND`/`UNSUPPORTED` при живой проверке → карантин в
   `02_processing/verify/rejected/`, чистка связей, запись в `known_problems.md`.
2. `WEAK`/`EVIDENCE_MISMATCH` → confidence≤0.80 + флаг.
3. Пересобрать индекс; `validate_graph.py` 0 ошибок.
4. Пересчитать состав графа и реальную rejection-rate по темам в `scale_quality_batch1.md`
   (перезаписать фиктивные 0%).

---

## Блок D — Недобор фактов по темам
1. Per-topic подсчёт верифицированных фактов после реконсиляции против цели ≥10/тему.
2. `[GUARD]` Если меньше — указать причину (реальная нехватка Tier-A по теме = допустимо) ИЛИ
   дособрать через A02 под живым гейтом. Не оставлять расхождение незадокументированным
   (исправляет «deltas: none» при явном недоборе).

---

## Блок E — Проверка кросс-темных связей
Каждую из 15 связей (особенно `retinol interacts_with glycolic_acid/niacinamide/SAP`) сверить:
есть ли реальный источник + `evidence_level`. `[GUARD]` Связь без подтверждённого источника —
удалить, не «логично же». Зафиксировать удалённые.

---

## Блок F — Tamper-evidence гейта (чтобы answer-key не повторился)
1. Каждый факт в графе обязан нести провенанс верификации:
   `"verified_via":"live|cache", "verified_at":"<ts>", "source_title":"<≤15 слов>"`.
2. Расширить `validate_graph.py` (или новый `integrity_check.py`): факт без `verified_via` и
   `verified_at` — ошибка валидации. `verified_via:"cache"` допускается только при наличии
   соответствующего файла кэша с провенансом.
3. `[GUARD]` Production verify-путь логирует факт обращения к API (URL+статус) в `ops/logs/`.
