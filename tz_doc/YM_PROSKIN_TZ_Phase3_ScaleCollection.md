## Блок 1 — Батч тем (выбраны для стресс-теста онтологии)
Три темы, намеренно пересекающиеся с ретиноидами и между собой по алиасам/применению:
1. **AHA/BHA кислоты** (glycolic, lactic, mandelic, salicylic) — класс Exfoliants.
2. **Витамин C** (L-ascorbic acid + деривативы: SAP, MAP, ascorbyl glucoside) — стресс алиасинга
   «vitamin C = ascorbic acid».
3. **Ниацинамид** (= vitamin B3 = nicotinamide) — стресс алиасинга + кросс-применение `used_for`
   акне/барьер, пересечение с ретиноидами.

`[GUARD]` Только эти 3 темы. Не расширять список в этом спринте.

Целевая таксономия (цели сбора, НЕ утверждённые факты — заполняет пайплайн из ретривала):
сущности активов + родительский класс; связи `is_a`/`belongs_to`, `used_for`,
`contraindicated_for`, `interacts_with` (напр. ретиноид+AHA — раздражение). Конкретные
утверждения и PMID — только из реального сбора через гейт.

## Блок 2 — Прогон пайплайна (под автоматическим гейтом)
Для каждой темы: A01 source-discovery → A02 research-collect (обязателен PMID/DOI) →
A03 fact-extract → A04 dedupe-sort → A05 **verify_gate** (авто-reject брака) → A06 kb-graph →
A07 ontology-guard. Доступ к графу — только через retriever (P003). Caveman-режимы по ролям.

`[GUARD]` Гейт Phase 2.5 обязателен на каждом факте. Никаких ручных обходов.

---

## Блок 3 — Межтемная онтология (ядро спринта, A07 глобально)
1. **Глобальный реестр сущностей** — A07 проходит по ВСЕМ темам, не по одной:
   - один компонент = одна каноническая сущность с aliases (ascorbic acid ≡ vitamin C;
     niacinamide ≡ vitamin B3 ≡ nicotinamide). `[GUARD]` Нет дублей одного вещества между темами.
2. **Кросс-темные связи** — создавать только реальные, подтверждённые источником:
   - `interacts_with` (напр. ретиноид + AHA);
   - общий `used_for` (ниацинамид и ретиноиды → acne) — как отдельные связи, не слияние сущностей.
   `[GUARD]` Не выдумывать связи; каждая `relationship` несёт `sources[]` и `evidence_level`.
3. **Контроль коллизий ID** — расширить `naming.md`/проверку: новые `kb_id` не конфликтуют с
   существующими; ретинол из Phase 1 не дублируется при пересечении тем.

---

## Блок 4 — Качество и пропускная способность
1. **Rejection-rate по теме** — после прогона зафиксировать % отбракованных фактов на тему в
   `ops/logs/scale_quality_batch1.md`. `[GUARD]` Если по теме >30% reject — пауза, разбор причины
   в `known_problems.md`, не продолжать молча (это сигнал деградации сбора/источников).
2. **eutils throughput** — на батче объём растёт; включить кэш (готов) + рассмотреть NCBI
   API-ключ (без ключа ~3 req/s). `[НЕТ ДАННЫХ]` ключ — если нет, троттлить и зафиксировать.
3. **Идемпотентность/резюмируемость** — пайплайн при повторном прогоне или после рестарта IDE
   не дублирует уже собранное; статус тем вести в `00_PROJECT_STATE/active_tasks.md`.

## Блок 5 — Definition of Done
- [ ] 3 темы собраны под гейтом; каждый факт прошёл verify_gate (лог).
- [ ] Per-topic acceptance: ≥1 класс + активы на тему; ≥10 верифицированных фактов на тему
      (или меньше с зафиксированной причиной — P007, не добивать мусором).
- [ ] A07 глобально: 0 дублей одного вещества между темами; aliases слиты
      (ascorbic acid/vit C, niacinamide/vit B3).
- [ ] Кросс-темные связи созданы только с источником+evidence; ≥1 `interacts_with`.
- [ ] `scale_quality_batch1.md`: rejection-rate по каждой теме; >30% → отражено в known_problems.
- [ ] Индекс пересобран; `validate_graph.py` 0 ошибок на расширенном графе.
- [ ] retriever возвращает корректные подграфы по каждой теме + рабочий кросс-темный запрос
      (напр. «acne» → ретиноиды + ниацинамид).
- [ ] `walkthrough.md` по шаблону; коммит + тег `phase-3-scale-batch1`.
