Детализация: Идеальный шаблон ТЗ для Claude Code
Ниже представлен готовый к копированию и внедрению шаблон системного ТЗ. Переменные в фигурных скобках {LIKE_THIS} заполняются под конкретную задачу.

Markdown
# ТЕХНИЧЕСКОЕ ЗАДАНИЕ: {НАЗВАНИЕ_ЗАДАЧИ} (Спецификация v{X.X})

## Слой 1: Предварительная проверка и безопасность (Pre-flight Git Safety)
Перед чтением кода или выполнением каких-либо изменений ОБЯЗАТЕЛЬНО выполни следующие шаги в терминале:
1. Выполни команду `git status`.
2. Проверь наличие незакоммиченных изменений в критических компонентах проекта:
   - В папке `ops/scripts/` и `ops/tests/`.
   - В конфигурационных файлах и библиотеках проекта.
3. Если обнаружены незакоммиченные изменения, которые могут повлиять на стабильность проекта или быть случайно перезаписаны, СТОП. Выполни пре-коммит сохранения текущего состояния:
   ```bash
   git add .
   git commit -m "pre-flight: savepoint before task {ID} - dirty state backup"
Если репозиторий чист, зафиксируй это в логе текущей сессии и переходи к Слою 2.

Слой 2: Контекст и Исходное состояние (Baseline)
Прочитай текущий статус проекта из 00_PROJECT_STATE/current_state.md, 00_PROJECT_STATE/HANDOFF_SNAPSHOT.md и 00_PROJECT_STATE/active_tasks.md.

Учти текущие ограничения и известные проблемы, описанные в known_problems.md.

Машиночитаемый источник истины (SSOT): Конфигурация пайплайна находится в 00_governance/pipeline_config.json. Не дублируй логику жестким кодированием (hardcode).

Слой 3: Суть инженерной задачи & Требования
Цель: {Четкое техническое описание целевого состояния системы}.

Входные данные: {Что подается на вход, инварианты данных}.

Бизнес-логика / Алгоритм:

{Шаг 1}

{Шаг 2}

Релиз-гейты и Критерии приемки:

{Например: Сохранение метрик SSIM или выполнение топологических инвариантов}.

Слой 4: Пошаговый протокол реализации (Execution Loop)
Реализуй задачу итеративно:

Внеси минимально необходимые изменения в целевые скрипты.

Проверь синтаксис и отсутствие регрессии.

Запусти существующий тестовый пакет для проверки стабильности:

Bash
python -m pytest ops/tests/  # или специфичные тесты проекта
Слой 5: Полное завершение и Самообзор (Self-Review & State Sync)
Задача считается выполненной ТОЛЬКО тогда, когда проведен ревью-анализ изменений и синхронизировано состояние проекта в базе знаний 00_PROJECT_STATE.
Выполни строго по порядку:

Проведи рефлексию проделанной работы. Определи, какие компоненты изменились, какие тесты пройдены, и остались ли открытые вопросы.

ОБЯЗАТЕЛЬНО ОБНОВИ / ПЕРЕЗАПИШИ файл 00_PROJECT_STATE/walkthrough.md, используя следующую структуру:

Статус выполнения задачи.

Детализированный список внесенных изменений.

Результаты верификации (логи, таблицы запусков, метрики до/после).

Открытые вопросы или обнаруженные баги (если есть).

Если задача меняет общие метрики или состав активных движков, обнови соответствующие разделы в 00_PROJECT_STATE/current_state.md и переведи задачу в статус выполненных в 00_PROJECT_STATE/active_tasks.md.

Слой 6: Финальная фиксация (Post-execution Commit)
После успешного обновления всех файлов состояния в 00_PROJECT_STATE, зафиксируй результат работы в Git атомарным коммитом:

Bash
git add 00_PROJECT_STATE/
git add ops/  # и другие измененные исходные файлы
git commit -m "feat/fix: completed task {ID} - {Краткое описание} | State Synced in walkthrough.md"

### Рекомендации

1. **Строгая изоляция контекста**: Использование команды `git status` на первом шаге предотвращает классическую проблему ИИ-агентов — случайную модификацию смежных файлов из-за неверной интерпретации путей.
2. **Использование табличной верификации**: Требуйте от Claude заполнения markdown-таблиц в `walkthrough.md` по аналогии с зафиксированными батчами (`COMP-022`, `COMP-028` и т.д.). Это заставляет модель подставлять реальные терминальные логи вместо их генерации "из головы".
3. **Автоматизация через CLI-флаги Claude Code**: При запуске утилиты передавайте ТЗ в качестве изолированного контекста (`claude "task instructions"`), ссылаясь на этот шаблон, чтобы агент не выходил за рамки инструкций.

### Риски

| Риск | Описание | Последствия | Метод предотвращения |
| :--- | :--- | :--- | :--- |
| **Затирание легаси-кода** | Модель модифицирует работающий инструмент (`deconstruct.py` или `compile_svg.py`) без учета локальных правок пользователя. | Сложная ручная отладка, потеря рабочих веток. | **Слой 1**: Принудительный `pre-flight` коммит. |
| **Галлюцинация успешного выполнения** | Модель рапортует об успехе, но не запускает тесты `ops/tests/`. | Скрытые синтаксические и логические ошибки в мастере. | **Слой 4 + 5**: Запрет закрытия задачи без логов тестов в `walkthrough.md`. |
| **Рассинхронизация SSOT** | Код изменен, но `current_state.md` и `pipeline_config.json` содержат устаревшие данные о версии спецификации (например, `3.5`). | Полная потеря прозрачности разработки для последующих сессий агента. | **Слой 5**: Фиксация обновления `walkthrough.md` как главного блокирующего гейта. |

### Альтернативы

Если Claude Code работает в режиме интерактивного терминала, альтернативным подходом является использование **Bash-скрипта-обертки** (Wrapper Script), который запускается на стороне хоста перед передачей задачи агенту. 

Скрипт проверяет состояние репозитория:
```bash
if [ -n "$(git status --porcelain)" ]; then
  git add . && git commit -m "auto-backup: pre-task state rescue"
fi
